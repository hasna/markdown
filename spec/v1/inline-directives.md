# OMP v1 -- Inline Directives Reference

Inline directives use the `{{content}}` syntax and can appear anywhere inside card headers or card bodies. They are resolved after all `@` directives have been processed but before cards are sent to the execution LLM.

## Syntax

```
{{content}}
```

### Regex

```
/\{\{(.+?)\}\}/g
```

Capture group 1: the directive content (trimmed).

The regex is non-greedy (`+?`) to handle multiple inline directives on the same line:

```
The secret is {{random(32, alphanumeric)}} and the id is {{uuid}}
```

This matches twice: `random(32, alphanumeric)` and `uuid`.

---

## Classification

Every inline directive is classified into one of two categories:

1. **Deterministic (built-in tool)** -- The content matches a known built-in tool pattern. The script resolves it directly with no LLM call. The result is always the same (or random but computed locally).

2. **Probabilistic (LLM prompt)** -- The content does not match any built-in tool. The script sends it to a cheap LLM with context from the surrounding card. The result is natural language generated by the LLM.

Classification regex (checked in order):

```
/^random\((.+)\)$/           → deterministic: random
/^uuid$/                     → deterministic: uuid
/^timestamp$/                → deterministic: timestamp
/^timestamp:(.+)$/           → deterministic: timestamp with format
/^env:(\w+)$/                → deterministic: env
/^ref:([\w.-]+)$/            → deterministic: ref
/^count:(\w+)$/              → deterministic: count
/^index$/                    → deterministic: index
/^hash:(\w+):(.+)$/          → deterministic: hash
```

If none match, the content is an LLM prompt.

---

## Built-in Tools (Deterministic)

These are resolved by the script without calling any LLM. They produce a concrete value every time.

---

### {{random(length, charset)}}

Generates a cryptographically random string of the specified length using the specified character set.

#### Syntax

```
{{random(32, alphanumeric)}}
{{random(16, hex)}}
{{random(24, base64)}}
{{random(8, numeric)}}
{{random(12, alpha)}}
```

#### Regex

```
/^random\((\d+),\s*(\w+)\)$/
```

Capture group 1: length (integer).
Capture group 2: charset name.

#### Charsets

| Charset | Characters |
|---------|------------|
| alphanumeric | A-Z, a-z, 0-9 |
| alpha | A-Z, a-z |
| numeric | 0-9 |
| hex | 0-9, a-f |
| base64 | A-Z, a-z, 0-9, +, / |

#### What the script does

Uses `crypto.randomBytes()` (Node.js) or equivalent to generate random bytes, then maps them to the specified charset. The result is a string of exactly the requested length.

#### Example

```
type: auth
id: auth
method: email+password
library: lucia-auth
hash: argon2
session: cookie

The session secret is {{random(64, alphanumeric)}}. Store it in the SESSION_SECRET environment variable.
```

Resolves to something like: `The session secret is k9Xm2pL7nQ4wR8jY1cB6vH3tA5dF0gE...`

---

### {{uuid}}

Generates a UUID v4 (random).

#### Syntax

```
{{uuid}}
```

#### Regex

```
/^uuid$/
```

No capture groups.

#### What the script does

Calls `crypto.randomUUID()` or equivalent. Returns a standard UUID v4 string in the format `xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx`.

#### Example

```
type: seed
id: seed
depends: db-push

Insert the admin user with id {{uuid}} and email andrei@hasna.com.
```

Resolves to: `Insert the admin user with id a1b2c3d4-e5f6-4a7b-8c9d-0e1f2a3b4c5d and email andrei@hasna.com.`

---

### {{timestamp}}

Returns the current date and time in ISO 8601 format.

#### Syntax

```
{{timestamp}}
```

#### Regex

```
/^timestamp$/
```

No capture groups.

#### What the script does

Returns `new Date().toISOString()`, producing a string like `2026-03-22T14:30:00.000Z`.

#### Example

```
type: deploy
id: deploy-prod
provider: vercel

Deploy to production. Build started at {{timestamp}}. Tag the deployment with this timestamp for rollback identification.
```

Resolves to: `Deploy to production. Build started at 2026-03-22T14:30:00.000Z.`

---

### {{timestamp:format}}

Returns the current date and time in a custom format.

#### Syntax

```
{{timestamp:YYYY-MM-DD}}
{{timestamp:HH:mm:ss}}
{{timestamp:YYYY-MM-DD HH:mm}}
{{timestamp:YYYYMMDD}}
```

#### Regex

```
/^timestamp:(.+)$/
```

Capture group 1: the format string.

#### Format tokens

| Token | Meaning | Example |
|-------|---------|---------|
| YYYY | 4-digit year | 2026 |
| MM | 2-digit month (01-12) | 03 |
| DD | 2-digit day (01-31) | 22 |
| HH | 2-digit hour (00-23) | 14 |
| mm | 2-digit minute (00-59) | 30 |
| ss | 2-digit second (00-59) | 45 |

#### What the script does

Gets the current date, then replaces each token in the format string with the corresponding value.

#### Example

```
type: migration
id: migration-create
depends: users, notes

Create a migration file named `{{timestamp:YYYYMMDD}}_initial_schema.sql` containing all table definitions.
```

Resolves to: `Create a migration file named 20260322_initial_schema.sql containing all table definitions.`

### {{env:VAR_NAME}}

Reads an environment variable. If the variable is not set, the script raises an error and halts -- it does not substitute an empty string or a default.

#### Syntax

```
{{env:DATABASE_URL}}
{{env:SESSION_SECRET}}
{{env:NODE_ENV}}
```

#### Regex

```
/^env:(\w+)$/
```

Capture group 1: the environment variable name (uppercase with underscores).

#### What the script does

1. Reads `process.env[VAR_NAME]`.
2. If the value is `undefined`, raises an error: `Environment variable VAR_NAME is not set`.
3. If the value is an empty string, returns the empty string (this is valid -- an explicitly set empty value is different from unset).

#### Example

```
type: database
id: db
engine: postgresql
orm: drizzle
url: {{env:DATABASE_URL}}

Connect to the PostgreSQL database using the connection string from the DATABASE_URL environment variable.
```

Resolves to: `url: postgres://user:pass@localhost:5432/noteapp` (or whatever the env var contains).

---

### {{ref:card-id.key}}

Reads a header value from another card in the document. This enables cards to reference each other's structured data without hardcoding values.

#### Syntax

```
{{ref:db.engine}}
{{ref:init.framework}}
{{ref:auth.method}}
{{ref:users.db}}
```

#### Regex

```
/^ref:([\w-]+)\.([\w-]+)$/
```

Capture group 1: the target card id.
Capture group 2: the header key to read.

#### What the script does

1. Looks up the card with the given id in the parsed card registry.
2. If the card is not found, raises an error: `Card 'card-id' not found`.
3. Reads the specified header key from that card.
4. If the key is not found on the card, raises an error: `Key 'key' not found on card 'card-id'`.
5. Returns the value as a string.

#### Example

```
type: migration
id: db-push
depends: users, notes, tags, note-tags

Push the schema to the {{ref:db.engine}} database at {{ref:db.file}} using {{ref:db.orm}}-kit push.
```

Resolves to: `Push the schema to the sqlite database at data/notes.db using drizzle-kit push.`

---

### {{count:type}}

Counts how many cards of a given type exist in the document.

#### Syntax

```
{{count:endpoint}}
{{count:table}}
{{count:page}}
{{count:test}}
```

#### Regex

```
/^count:(\w+)$/
```

Capture group 1: the card type to count.

#### What the script does

Iterates through all parsed cards and counts those whose `type` header matches the given value. Returns the count as a string (e.g., `"8"`).

#### Example

```
type: test
id: api-coverage
pattern: src/api/__tests__/*.test.ts

Write integration tests for all {{count:endpoint}} API endpoints. Each endpoint needs at least a success case, an auth-failure case, and a validation-error case.
```

If the document has 8 endpoint cards, resolves to: `Write integration tests for all 8 API endpoints.`

---

### {{index}}

Returns the current iteration index inside an `@repeat` block. Zero-based. This directive is only valid inside a card that was produced by `@repeat`.

#### Syntax

```
{{index}}
```

#### Regex

```
/^index$/
```

No capture groups.

#### What the script does

Returns the current iteration number as a string. If used outside an `@repeat` context, raises an error: `{{index}} can only be used inside @repeat`.

#### Example

```
@repeat 4
type: endpoint
id: shard-{{index}}
method: GET
path: /api/shard/{{index}}/health
auth: none

Check the health of shard {{index}}. Return the shard's connection count and memory usage.
```

Produces four cards. The third one (index 2) resolves to:
```
type: endpoint
id: shard-2
method: GET
path: /api/shard/2/health
auth: none

Check the health of shard 2. Return the shard's connection count and memory usage.
```

### {{hash:algorithm:input}}

Hashes a string value using the specified algorithm. Useful for generating deterministic identifiers from card data.

#### Syntax

```
{{hash:sha256:my-secret-value}}
{{hash:md5:session-key-prefix}}
{{hash:sha256:{{ref:init.name}}-{{timestamp:YYYYMMDD}}}}
```

#### Regex

```
/^hash:(\w+):(.+)$/
```

Capture group 1: the hash algorithm (sha256, sha512, md5).
Capture group 2: the input string to hash.

#### Supported algorithms

| Algorithm | Output length (hex) |
|-----------|-------------------|
| md5 | 32 characters |
| sha256 | 64 characters |
| sha512 | 128 characters |

#### What the script does

1. Takes the input string (which may itself contain resolved inline directives -- they are resolved first via inner-to-outer evaluation).
2. Creates a hash using the specified algorithm.
3. Returns the hex-encoded digest.

#### Example

```
type: deploy
id: deploy-prod
provider: vercel

Tag this deployment with build hash {{hash:sha256:{{ref:init.name}}-{{timestamp:YYYYMMDD}}}}. Use the first 8 characters as the short tag.
```

The inner directives resolve first: `ref:init.name` becomes `NoteApp`, `timestamp:YYYYMMDD` becomes `20260322`. Then the hash input is `NoteApp-20260322`, which is SHA-256 hashed. The result looks like: `Tag this deployment with build hash a3f7c9e1...`

## LLM Directives (Probabilistic)

Any `{{content}}` that does not match a built-in tool pattern is sent to a cheap LLM for resolution. The LLM generates a response based on the content and the surrounding card context.

### How the prompt is built

The script constructs a prompt with these components:

1. **System instruction**: A fixed preamble telling the LLM its role.
2. **Card context**: The card's type, id, and all header key-value pairs.
3. **Surrounding text**: The paragraph containing the `{{}}` directive.
4. **Related cards**: Other cards referenced in `depends` or sharing the same `db`/`path` prefix.
5. **The directive content**: The text inside `{{}}`.

#### Prompt template

```
You are filling a placeholder in an OMP document.

Card type: {card.type}
Card id: {card.id}
Structured data already extracted:
{all card headers as key: value lines}

Related cards:
{headers of depends cards and related cards}

Surrounding context:
"{the paragraph containing the {{}} directive}"

Your task: {the content inside {{}}}

Respond with ONLY the answer. No explanation, no markdown formatting, no quotes. Just the raw value.
```

### Why related cards are included

The LLM often needs context beyond the current card. For example, when generating seed data for a notes app, the LLM needs to know the table schema (column names, types) to produce realistic data. By including the headers of related cards (tables, endpoints the card depends on), the LLM can produce contextually accurate results.

### Examples of LLM directives

Each example shows the directive, the card it appears in, and what the LLM might produce.

---

#### Generating realistic sample data

```
type: seed
id: seed
admin-email: andrei@hasna.com
sample-notes: 5
depends: db-push

Create 5 sample notes with these titles: {{generate 5 realistic note titles for a productivity app}}
```

The LLM receives the card context (seed type, depends on db-push), related cards (the table definitions from db-push's dependencies), and the directive.

LLM might respond: `Meeting Notes, Project Ideas, Weekly Goals, Book Recommendations, Travel Plans`

---

#### Choosing an icon

```
type: component
id: empty-state
props: message (string), icon (string)
file: src/components/EmptyState.tsx
depends: init

Use the {{suggest the best icon name from lucide-react for a notes feature}} icon from lucide-react. Display it above the message at 48px size with muted color.
```

LLM might respond: `NotebookPen`

#### Generating a technical strategy

```
type: endpoint
id: search-notes
method: GET
path: /api/notes/search
auth: required
params: q (string, required)
depends: auth

{{describe an efficient full-text search strategy for SQLite}} -- implement this approach for searching notes by title and content.
```

LLM might respond: `Use SQLite's FTS5 extension. Create a virtual table notes_fts with tokenize='porter' covering the title and content columns. On note insert/update, sync the FTS table. For search queries, use the MATCH operator with rank ordering for relevance.`

```
type: page
id: notes-list
path: /notes
auth: required
depends: root-layout, list-notes

When the user has no notes, show an empty state with the message "{{write a friendly empty state message for when there are no notes}}".
```

LLM might respond: `No notes yet. Tap the + button to capture your first thought.`

#### Suggesting a color palette

```
type: component
id: tag-chip
props: name (string), color (string)
file: src/components/TagChip.tsx

Display the tag name in a small rounded badge. Use these background colors for built-in tags: {{suggest 5 distinct pastel hex colors suitable for tag badges on a white background}}.
```

#### Generating a validation message

```
type: endpoint
id: create-note
method: POST
path: /api/notes
auth: required
body: { title: string, content: string }

Validate the request body. If title is missing, return 422 with message "{{write a concise validation error message for a missing note title}}". If content is missing, return 422 with message "{{write a concise validation error message for missing note content}}".
```

LLM might respond (for the first): `Title is required`
LLM might respond (for the second): `Content cannot be empty`

```

file: src/middleware/rate-limit.ts
depends: init

Apply rate limiting to all API routes matching {{suggest a regex pattern that matches all /api/* routes but excludes /api/health}}.
```

LLM might respond: `^\/api\/(?!health).*$`

## Resolution Order

Inline directives are resolved in this order:

1. **Inner directives first** -- If a `{{}}` contains another `{{}}`, the innermost is resolved first. Example: `{{hash:sha256:{{ref:init.name}}}}` first resolves `{{ref:init.name}}` to `NoteApp`, then resolves `{{hash:sha256:NoteApp}}`.

2. **Deterministic before probabilistic** -- All built-in tools are resolved before any LLM calls are made. This ensures LLM prompts can reference already-resolved values.

3. **Top-to-bottom within a card** -- Within a single card, directives are resolved in the order they appear.

4. **Card execution order** -- Cards earlier in the DAG execution order have their directives resolved first. A card can reference values from cards that have already been executed.

## Error Handling

| Error | Cause | Resolution |
|-------|-------|------------|
| Environment variable not set | `{{env:VAR}}` and `VAR` is undefined | Set the variable or remove the directive |
| Card not found | `{{ref:missing-id.key}}` | Check the card id exists in the document |
| Key not found on card | `{{ref:card.missing-key}}` | Check the header key exists on the target card |
| Index outside @repeat | `{{index}}` used outside a `@repeat` block | Move the directive inside an `@repeat` card or remove it |
| Unknown hash algorithm | `{{hash:blake2:input}}` | Use a supported algorithm: md5, sha256, sha512 |
| Invalid random charset | `{{random(16, emoji)}}` | Use a supported charset: alphanumeric, alpha, numeric, hex, base64 |
| LLM returned empty | The LLM produced no output for a probabilistic directive | Retry once; if still empty, raise an error |
| Nested resolution failure | An inner directive failed, so the outer one cannot resolve | Fix the inner directive first |

## Best Practices

### When to use deterministic directives

Use built-in tools whenever the value can be computed without judgment:

- Secrets and tokens: `{{random(64, alphanumeric)}}`
- Identifiers: `{{uuid}}`
- Timestamps: `{{timestamp:YYYY-MM-DD}}`
- Cross-references: `{{ref:db.engine}}`
- Metrics: `{{count:endpoint}}`

### When to use LLM directives

Use LLM directives when the value requires creative judgment or domain knowledge:

- UI copy and messages
- Sample data that should feel realistic
- Technical strategy suggestions
- Icon or color choices
- Validation error wording
- Naming suggestions

### Keep LLM directives focused

Good -- asks for one specific thing:
```
{{generate 5 realistic note titles for a productivity app}}
```

Bad -- too broad, will produce inconsistent results:
```
{{describe the entire UI for the notes page including layout, colors, interactions, and animations}}
```

### Provide context through card headers

The more structured data the card has in its headers, the better the LLM directive results. The LLM uses headers as context:

```
type: seed
id: seed
admin-email: andrei@hasna.com
sample-notes: 5
sample-tags: [work, personal, ideas, urgent, reference]
depends: db-push, auth

Create notes with titles: {{generate 5 realistic note titles for a productivity app}}
```

The LLM sees the tags list and can generate titles that relate to those tags (e.g., "Project Ideas" for the `ideas` tag).

### Avoid side effects in directives

Inline directives should produce a value, not trigger an action. Use card bodies and `call` card types for actions.

Good:
```
The API key is {{env:API_KEY}}
```

Bad (do not do this):
```
{{create a new file at src/config.ts with the database configuration}}
```

File creation is a card-level action, not an inline directive.
